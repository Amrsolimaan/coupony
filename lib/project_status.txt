📚 BASEREPOSITORY - UPDATED API:
=================================================================

ORIGINAL METHOD (Still Available):
-----------------------------------
Future<Either<Failure, T>> fetchWithCacheStrategy<T>({
  required Future<T> Function() remoteCall,
  required Future<T> Function() localCall,
  required Future<void> Function(T data) cacheCall,
  bool shouldCache = true,
  Future<bool> Function()? cacheValidation,
});

NEW METHOD (Recommended):
-------------------------
Future<Either<Failure, T>> fetchWithAutoCache<T>({
  required Future<T> Function() remoteCall,
  required String cacheKey,
  required String boxName,
  Duration? cacheTTL,
  bool forceRefresh = false,
});

EXAMPLE USAGE:
--------------
class CouponRepositoryImpl extends BaseRepository {
  @override
  Future<Either<Failure, List<Coupon>>> getCoupons() async {
    return fetchWithAutoCache<List<CouponModel>>(
      remoteCall: () => remoteDataSource.getCoupons(),
      cacheKey: StorageKeys.couponsList,
      boxName: StorageKeys.couponsBox,
      cacheTTL: AppConstants.couponsCacheDuration,
      forceRefresh: false,  // Set true for pull-to-refresh
    );
  }
}

💡 IMPLEMENTATION WISDOM:
=================================================================

CACHE USAGE PATTERNS:
----------------------
1. **Frequently changing data** (Coupons):
   - Short TTL (15 minutes)
   - Background updates
   - Force refresh on pull-to-refresh

2. **Stable data** (Stores, Categories):
   - Longer TTL (1 hour - 7 days)
   - Reduces API calls
   - Saves battery/data

3. **Media files**:
   - 7-day TTL
   - Quota-managed
   - Show instantly when cached

4. **User data**:
   - 1-day TTL
   - Secure storage for tokens
   - Hive for profile info

--------------------
