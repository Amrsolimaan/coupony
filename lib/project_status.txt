ORIGINAL METHOD (Still Available):

Future<Either<Failure, T>> fetchWithCacheStrategy<T>({
  required Future<T> Function() remoteCall,
  required Future<T> Function() localCall,

  bool shouldCache = true,
  Future<bool> Function()? cacheValidation,

NEW METHOD (Recommended):
-------------------------
Future<Either<Failure, T>> fetchWithAutoCache<T>({
  required Future<T> Function() remoteCall,
  required String cacheKey,
  required String boxName,
  Duration? cacheTTL,
  bool forceRefresh = false,
});

EXAMPLE USAGE:
--------------
class CouponRepositoryImpl extends BaseRepository {
  @override
  Future<Either<Failure, List<Coupon>>> getCoupons() async {
    return fetchWithAutoCache<List<CouponModel>>(
      remoteCall: () => remoteDataSource.getCoupons(),
      cacheKey: StorageKeys.couponsList,
      boxName: StorageKeys.couponsBox,
      cacheTTL: AppConstants.couponsCacheDuration,
      forceRefresh: false,  // Set true for pull-to-refresh
    );
  }
}

💡 IMPLEMENTATION WISDOM:
=================================================================

CACHE USAGE PATTERNS:
----------------------
1. **Frequently changing data** (Coupons):
   - Short TTL (15 minutes)
   - Background updates
   - Force refresh on pull-to-refresh

2. **Stable data** (Stores, Categories):
   - Longer TTL (1 hour - 7 days)
   - Reduces API calls
   - Saves battery/data

3. **Media files**:
   - 7-day TTL
   - Quota-managed
   - Show instantly when cached

4. **User data**:
   - 1-day TTL
   - Secure storage for tokens
   - Hive for profile info

- Rebuilt search bar as a single unified `Container` (width: `double.infinity`, height: `55.h`).
- `BoxDecoration` (border, shadow, color) applied to the outer container only.
- Icons (Search, Mic, Back) rendered inside a `Row` on top of the container layer.

- No input formatters — full Arabic character support.
- Search and Back icons wrapped in `GestureDetector` and linked to their functions.

## ✅ RESOLVED: Navigator Assertion Error (Red Screen Crash)
- Root Cause: `location_map_page.dart` was calling `context.go(AppRouter.permissionNotificationIntro)`
  inline inside the "Confirm Location" button's `onPressed`, immediately after calling
  `cubit.confirmLocation()`. The cubit simultaneously emitted a `navSignal` which triggered
  the `PermissionFlowWrapper`'s own listener — resulting in two concurrent GoRouter pushes
  on the same navigator, causing the `Failed assertion: 'route._navigator == navigator'` crash.
- Fix Applied:
  1. Removed the inline `context.go()` call from the confirm button's `onPressed`.
  2. Added a `BlocListener` directly inside `LocationMapPage.build()` that listens for
     `PermissionNavigationSignal` values and calls `context.go()` only from there.
  3. `clearNavigationSignal()` is called immediately after each navigation to prevent
     re-triggering on rebuilds.
  4. All async callbacks in the page already guard with `if (!mounted) return`.
- Result: Navigation from Location Map → Notification Intro is now crash-free.

## ✅ COMPLETED: Map Loading UX (Phase 3)
- `_isMapLoading` flag initialized to `true` on page load.
- A full-screen `CircularProgressIndicator` overlay is shown until the map is ready.
- `onMapCreated` now calls `setState(() => _isMapReady = true)` to mark controller readiness.
- `onCameraIdle` hides the overlay (with a 500ms buffer) only after `_isMapReady` is true,
  ensuring tiles have finished rendering before the overlay disappears.
- `if (!mounted)` guards added to all delayed callbacks.
